# Priority Queue (`@std.heap`) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a persistent priority queue under `@std.heap`, implemented as a pairing heap, so workloads that repeatedly pop the highest-priority element (Dijkstra, scheduling, top-k, k-way merge) have an efficient immutable collection.

**Architecture:** Library-first, mirroring `@std.queue`: no compiler or runtime primitive, entirely Twinkle stdlib code composed over `Vector`. The priority-queue ADT is implemented as a **pairing heap** — a tree of nodes where each node's value orders before all of its children. Unlike an array-backed binary heap, a pairing heap never mutates a slot by index, so insert/merge/pop produce new nodes with structural sharing and stay genuinely persistent. The comparator is stored once in the heap record (so `push`/`pop`/`merge` need no comparator argument and a min-heap vs max-heap is just a different comparator at construction).

**Tech Stack:** Twinkle stdlib (`boot/stdlib/heap.tw`), the `fn(T, T) Order` comparator convention and `<T: Ord>` contract bound (both already used by `Vector.sort_by` / `Vector.sort` in `boot/prelude/vector.tw`), boot test suites, and `tools/generate_core_lib.py` (auto-run by the Makefile).

## Design decisions

These are chosen defaults; flag in review if you want them changed before execution.

- **Module name `@std.heap`, type `Heap<T>`.** Concise and conventional (Python `heapq`, Rust `BinaryHeap`). Docs open by naming it a priority queue so the ADT is discoverable. File is `boot/stdlib/heap.tw`, paralleling `boot/stdlib/queue.tw`.
- **Priority order is "compares first."** The element for which the comparator returns `.Lt` against all others comes out first. With `Int.compare` that is a **min-heap**; pass a reversed comparator (`fn(a, b) { b.compare(a) }`) for a max-heap. `peek` / `pop` (not `peek_min`) name it, because "min" is comparator-relative.
- **Comparator stored in the heap record** (`cmp: fn(T, T) Order`). Record fields may hold closures — this is the same shape as the capability records in `CLAUDE.md` (`type Show<T> = .{ to_string: fn(T) String }`). Storing it avoids passing a comparator to every op and prevents callers from mixing mismatched comparators between `push` and `pop` (which would corrupt heap order).
- **`size` cached in the record** for O(1) `len`, maintained by `push`/`pop`/`merge`.
- **Tree as an enum** (`{ Empty, Node(T, Vector<Tree<T>>) }`) for idiomatic pattern matching, mirroring how `Option` / `Shape` are modeled.

## Public API

```tw
use @std.heap
use @std.heap.{Heap}

// min-heap over Ints via an explicit comparator
h: Heap<Int> = heap.new(fn(a: Int, b: Int) Order { a.compare(b) })
h = h.push(3).push(1).push(2)

case h.pop() {
  .Some(p) => {
    value := p.value   // 1
    h = p.rest
  },
  .None => {},
}
```

Types:

```tw
pub type Heap<T> = .{
  root: Tree<T>,
  cmp: fn(T, T) Order,
  size: Int,
}

pub type Tree<T> = { Empty, Node(T, Vector<Tree<T>>) }

pub type Pop<T> = .{
  value: T,
  rest: Heap<T>,
}
```

`Tree` is a representation detail exposed publicly (Twinkle records/types are public today, matching how `@std.queue` documents its `front`/`back` fields). Callers should use the module API and treat `root` as opaque.

Functions:

```tw
pub fn new<T>(cmp: fn(T, T) Order) Heap<T>
pub fn new_ord<T: Ord>() Heap<T>
pub fn singleton<T>(value: T, cmp: fn(T, T) Order) Heap<T>
pub fn from_vector<T>(xs: Vector<T>, cmp: fn(T, T) Order) Heap<T>
pub fn to_vector<T>(h: Heap<T>) Vector<T>

pub fn len<T>(h: Heap<T>) Int
pub fn is_empty<T>(h: Heap<T>) Bool

pub fn push<T>(h: Heap<T>, value: T) Heap<T>
pub fn peek<T>(h: Heap<T>) T?
pub fn pop<T>(h: Heap<T>) Pop<T>?
pub fn merge<T>(a: Heap<T>, b: Heap<T>) Heap<T>
```

Qualified and method-call forms both work (`heap.push(h, x)` ≡ `h.push(x)`); the receiver-first parameter order makes `h.push(...)`, `h.pop()`, `a.merge(b)` resolve as inherent methods.

```tw
/// Persistent priority queue implemented as a pairing heap.
///
/// A `Heap<T>` is a comparator plus a pairing-heap tree: every node's value
/// orders before (`.Lt`) all of its children. The element that orders before
/// all others is at the root, so `peek` / `pop` are cheap. The comparator is
/// stored in the heap, so `push` / `pop` / `merge` take no comparator argument
/// and the same heap is a min-heap or max-heap purely by how it was built
/// (`Int.compare` gives a min-heap; a reversed comparator gives a max-heap).
///
/// `Tree<T>` is a representation detail. Use the module API and treat `root`
/// as opaque.

pub type Heap<T> = .{ root: Tree<T>, cmp: fn(T, T) Order, size: Int }

pub type Tree<T> = { Empty, Node(T, Vector<Tree<T>>) }

pub type Pop<T> = .{ value: T, rest: Heap<T> }

/// Empty heap ordered by `cmp`. The element for which `cmp` returns `.Lt`
/// against all others is dequeued first.
pub fn new<T>(cmp: fn(T, T) Order) Heap<T> {
  Heap.{ root: .Empty, cmp, size: 0 }
}

/// Heap of one value.
pub fn singleton<T>(value: T, cmp: fn(T, T) Order) Heap<T> {
  Heap.{ root: .Node(value, []), cmp, size: 1 }
}

/// Number of values in the heap (O(1)).
pub fn len<T>(h: Heap<T>) Int {
  h.size
}

/// True when the heap has no values.
pub fn is_empty<T>(h: Heap<T>) Bool {
  h.size == 0
}

/// Highest-priority value (orders before all others), or `.None` if empty.
pub fn peek<T>(h: Heap<T>) T? {
  case h.root {
    .Empty => .None,
    .Node(value, _) => .Some(value),
  }
}
```

```tw
/// Combine two pairing-heap trees so the root orders before the other.
/// The losing tree becomes a child of the winning root.
fn meld<T>(cmp: fn(T, T) Order, a: Tree<T>, b: Tree<T>) Tree<T> {
  case a {
    .Empty => b,
    .Node(av, ac) => {
      case b {
        .Empty => a,
        .Node(bv, bc) => {
          case cmp(av, bv) {
            .Gt => .Node(bv, bc.append(a)),
            _ => .Node(av, ac.append(b)),
          }
        },
      }
    },
  }
}

/// Insert a value (amortized O(1)). Melds a single-node tree into the root.
pub fn push<T>(h: Heap<T>, value: T) Heap<T> {
  h.root = meld(h.cmp, h.root, .Node(value, []))
  h.size = h.size + 1
  h
}
```

```tw
/// Two-pass pairing merge over a node's children: meld adjacent pairs left to
/// right, then fold the results right to left into a single tree.
fn merge_pairs<T>(cmp: fn(T, T) Order, cs: Vector<Tree<T>>) Tree<T> {
  n := cs.len()
  if n == 0 {
    return .Empty
  }
  if n == 1 {
    return cs[0]
  }

  paired: Vector<Tree<T>> = []
  i := 0
  for i + 1 < n {
    paired = paired.append(meld(cmp, cs[i], cs[i + 1]))
    i = i + 2
  }
  if i < n {
    paired = paired.append(cs[i])
  }

  acc := paired[paired.len() - 1]
  j := paired.len() - 2
  for j >= 0 {
    acc = meld(cmp, paired[j], acc)
    j = j - 1
  }
  acc
}

/// Remove and return the highest-priority value, or `.None` if empty.
/// Amortized O(log n).
pub fn pop<T>(h: Heap<T>) Pop<T>? {
  case h.root {
    .Empty => .None,
    .Node(value, children) => {
      h.root = merge_pairs(h.cmp, children)
      h.size = h.size - 1
      .Some(Pop.{ value, rest: h })
    },
  }
}
```

```tw
/// Combine two heaps into one (amortized O(1)). Uses the left heap's
/// comparator; pass heaps built with the same ordering.
pub fn merge<T>(a: Heap<T>, b: Heap<T>) Heap<T> {
  a.root = meld(a.cmp, a.root, b.root)
  a.size = a.size + b.size
  a
```

```tw
/// Build a heap from a vector by inserting each element (O(n), since each
/// pairing-heap insert is amortized O(1)).
pub fn from_vector<T>(xs: Vector<T>, cmp: fn(T, T) Order) Heap<T> {
  h := new(cmp)
  for x in xs {
    h = h.push(x)
  }
  h
}

/// Drain the heap into a vector in priority order (heapsort, O(n log n)).
/// The input heap is not modified.
pub fn to_vector<T>(h: Heap<T>) Vector<T> {
  out: Vector<T> = []
  cur := h
  for cur.len() > 0 {
    case cur.pop() {
      .Some(p) => {
        out = out.append(p.value)
        cur = p.rest
      },
      .None => {},
    }
  }
  out
}
```

## Self-review

**Spec coverage:** Every public function in the API surface has an implementing task — `new`/`singleton`/`len`/`is_empty`/`peek` (Task 1), `push` (Task 2), `pop` (Task 3), `merge` (Task 4), `from_vector`/`to_vector` (Task 5), `new_ord` (Task 6) — plus docs and finalize (Task 7). Persistence (operations don't mutate inputs) is covered by the Task 3 "original unchanged" and Task 4 "inputs unchanged" tests. Min-heap and max-heap both covered (Task 6).

**Type consistency:** `Heap<T>` carries `root: Tree<T>`, `cmp: fn(T, T) Order`, `size: Int` throughout. `pop` returns `Pop<T>?` with fields `value` / `rest`, matching the `Pop` type and the suite helpers. `meld` and `merge_pairs` are private and take `(cmp, ...)` consistently. `peek` returns `T?`. All comparator parameters use `fn(T, T) Order`.

**Known risks / first checkpoint:** Task 1's `make bundle-cli` is the de-risking gate for the two unproven shapes — a closure-typed record field (`cmp`) and a self-recursive generic enum through `Vector` (`Tree<T>`). The capability-record pattern (`Show<T>`) and recursive enums (`Option`, `List`) are each used elsewhere, but their combination here compiles for the first time in Task 1. If it fails, fix the representation before building dependent ops. A secondary risk is whether `<T: Ord>` resolves through a stored-comparator constructor (Task 6); the `Vector.sort` precedent makes this low-risk.

## Deferred follow-ups

- `peek` / `pop` only expose the single highest-priority element; `pop_n` / top-k helpers can be added when a use case appears.
- Heap equality and `Stringify` are omitted initially (order of equal-priority elements is not specified, so structural `==` would be misleading). Add later only with a defined semantics.
- `iter()` is deferred in favor of `to_vector()` plus vector iteration, matching the `@std.queue` precedent.
- A real benchmark (build + drain vs `Vector.sort`, and a Dijkstra-style decrease-key-free workload) should confirm the amortized claims before any performance wording is strengthened.
